# 连接状态可视化 — 设计文档

## 背景与问题

当前前端（`server/static/index.html`）只能在**点击发送、撞上错误码的那一刻**才感知到状态。
用户无法在发送前主动看到：目标设备（`win-fukuoka` / `mac-china`）此刻到底连没连上；
出问题时也分不清是"服务端不可达"还是"客户端离线"。

信息其实已经存在——服务端内存里 `agents.websockets` 保存着每个 Agent 的活连接，
发送时的 `503`（客户端离线）与 `fetch` 异常（服务端不可达）也本就可区分。
缺的只是：①一个能**主动查询**在场状态的接口，②前端把三态**清晰可视化**。

## 第一性原理判断（为什么是这个方案）

- 这个需求本质是"**按需读取当前在场状态**"，不是"要实时推送"。因此 HTTP 轮询是它的
  最优表达，不是妥协。给发送方加 WebSocket 推送只为把"最多 4 秒延迟"变成"瞬时"，
  而人类在"发送前瞟一眼设备在不在线"的场景里感知不到这点差异，却要付出 WS 生命周期
  管理、服务端广播机制等真实复杂度——属于过度设计，明确不做。
- **目录结构不动**：状态功能与 `server/agent/docs` 的分层完全正交。
- **现有协议不动**：只**新增**一个只读状态接口（附加，不破坏既有 WS/HTTP 契约），
  并在 `docs/protocol.md` 中补充记录。
- 真正值得动协议的是"离线消息不丢的存储转发队列"，但那是**独立的另一个特性**，
  不在本轮范围，将来单开一轮设计。

## 范围

**做**：服务端新增 `GET /api/status`；前端轮询它并把三态可视化。
**不做**：消息队列/持久化、目录重构、发送方 WebSocket、设备清单可配置化。

## 组件设计

### 1. 服务端：`GET /api/status`

- **文件**：`server/app.py`
- **鉴权**：要求 `X-API-Key`，与 `/api/send` 一致；缺失或错误返回 `401`；
  `API_KEY` 未配置返回 `500`。复用现有的 `check_api_key()`。
- **返回**（`200`）：遍历 `DEVICES` 的每个 device_id，判断是否存在于
  `agents.websockets` 中，得到所有设备的在场布尔映射（键必须来自 `DEVICES`
  的全量列表，不能只用 `agents.websockets` 的 key——否则离线设备会从映射里
  消失，破坏"返回全部设备"这一前提）：

  ```json
  {"devices": {"win-fukuoka": true, "mac-china": false}}
  ```

- **为什么返回全部设备**：前端一次轮询即可同时更新"选中徐"和"下拉框每一项"，
  也天然供将来的安卓 APP 复用同一接口。
- **无副作用**：纯读内存，不改动任何连接状态。

### 2. 前端：三态可视化

- **文件**：`server/static/index.html`（保持单文件、内联 CSS/JS 的现状）
- **轮询策略**：仅在密钥非空时轮询；进页面查一次、切换设备查一次、之后每约 4 秒一次。
  失焦/隐藏时可暂停（可选优化，非必须）。
- **三态判定**：

  | 轮询结果 | 状态 | 文案 |
  |---|---|---|
  | `fetch` 抛异常 | 🔴 服务端不可达 | 无法连接到中继服务器 |
  | `401` | 🔴 密钥错误 | 密钥错误 |
  | `200` 且该设备为 `true` | 🟢 已连接 | （设备名）已连接 |
  | `200` 且该设备为 `false` | 🟡 客户端离线 | （设备名）未连接 |

- **两处展示**：
  - **选中设备状态徐**：下拉框下方一个带颜色圆点的 DOM 元素（颜色/圆点自由控制）。
  - **下拉框每一项**：选项文字后缀标注在线/离线，如「福冈 Windows（在线）」。
    - 平台约束：手机原生 `<option>` 不能可靠上色，但可改文字，因此下拉项用
      **文字后缀**而非颜色；颜色圆点只用于选中徐。

### 3. 发送逻辑

保持现状不变。状态显示是新增的主动感知层。选中设备即使已知离线也**仍允许发送**
（在场状态可能在两次轮询间变化），不加拦截，避免误判。

## 数据流

```
浏览器 --(每~4s, 带 X-API-Key)--> GET /api/status --> 读 agents.websockets
   <--(200 {"devices": {...}} | 401 | 网络异常)--
   --> 更新: 选中设备徐(带色圆点) + 下拉每一项文字后缀
```

## 错误处理

- 服务端：沿用 `check_api_key()` 的 `401`/`500`；`/api/status` 本身不会因设备离线报错，
  离线只是布尔值为 `false`。
- 前端：`fetch` 异常与 `401` 分别映射到"服务端不可达"与"密钥错误"两个明确文案，
  不混为一句"出错了"。

## 测试与验证

- **服务端**：新增 `server/tests/`（当前无测试，这是第一个，顺带建立测试脚手架）。
  用 FastAPI `TestClient` 覆盖：
  - 无密钥 / 错密钥 → `401`
  - 模拟 Agent 连上 `/ws/agent` 后，`/api/status` 中该设备为 `true`
  - 断开后为 `false`
- **前端**：vanilla JS 单文件、无测试框架，靠手动验证——本地起 server，
  连上/断开 Mac agent，观察选中徐与下拉项是否随之实时变化。

1. 从 `main` 拉专用分支 `feature/connection-status`（已完成）。

3. 审查：由 Codex 审查代码 → 按意见修改 → 再审，循环直到通过。
4. 提交：commit 信息用**中文**（保留 `Co-Authored-By` trailer）。

6. 用户确认无误并**明确许可**后，才合并回 `main`、删除该分支，再进入下一件事。

## 验收标准

- `GET /api/status` 无密钥/错密钥返回 `401`；正确密钥返回各设备在场布尔映射。
- Agent 连上/断开后，`/api/status` 的对应布尔值随之变化。
- 前端能明确区分并显示三态：服务端不可达 / 客户端离线 / 已连接。
- 前端下拉框每一项显示各自的在线/离线文字后缀；切换设备时选中徐实时刷新。
- 服务端新增测试全部通过。
- 发送流程行为不回归。
