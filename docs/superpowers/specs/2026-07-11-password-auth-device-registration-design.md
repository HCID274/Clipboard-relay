# 密码鉴权与设备自助注册 — 设计文档

## 背景与问题

当前系统用**单一全局 `API_KEY`**（一段很长的随机串）做鉴权：HTTP 接口靠 `X-API-Key`
校验，WebSocket `/ws/agent?device_id=xxx` 用同一把 key 校验，设备白名单
（`win-fukuoka` / `mac-china`）**硬编码**在 `server/app.py` 的 `DEVICES` 字典里。

两个痛点：

1. **长 token 难记、配置麻烦**：新设备接入要手动复制粘贴一长串 key，还要有人先在
   服务器代码里把设备名加进 `DEVICES` 字典并重新部署，做不到"一步到位"。
2. **无法自助接入**：来一台新 Mac / Windows，没有脚本能"装完就自动连上并出现在
   设备列表里"。

目标：**用好记的密码代替长 token**，并让**新设备跑个安装脚本就能自解释注册接入**，
同时对"密码泄露后被批量刷垃圾设备"做一层轻量防护。

**继承的既有限制（本轮不改变，明确记录避免被误认为新引入的问题）**：浏览器页面
把密钥明文存在 `localStorage`——这是现有行为，本轮沿用，不算新增风险面。

## 第一性原理判断（为什么是这个方案）

这一节记录设计过程中被逐条否掉的"更复杂方案"，说明为什么最终收敛到最简形态。

- **不需要 token 层**。密码每次随 TLS 请求发送、服务端直接比对即可。"登录换长期
  token"要多一套接口、多一份状态、多一条故障路径；在单人/几台设备场景下，一旦
  每台设备用的是共享密码，密码本身就已经在履行 API Key 的职责，再套一层 token
  属于过度设计。
- **不需要一次性配对码**。配对码防的是"密码泄露后被顺手注册新设备"，但这个风险在
  可信小规模场景可接受，且已由下面的"设备数上限"兜底，不值得为它增加一步手动输入。
- **不需要用户表 / 多用户隔离**。当前只有一个使用者，用户表是为将来"开放给别人用"
  预留的，属于 YAGNI。**多用户隔离明确推到下一轮**，真要开放时再引入账号体系。
- **不需要每台设备独立密码**。共享一套账号密码即可；唯一代价是"丢一台设备只能改
  共享密码、其余设备重新登录"，这个代价对个人几台设备完全可以接受，不值得为"单
  设备撤销"去维护设备密码表。
- **`device_id` 是非鉴权的路由标识**，不是凭据。它只回答"这条消息发给哪台设备"，
  鉴权完全由共享密码把关，因此设备可以自解释上报自己的名字。

## 范围

**做**：
- 全局 `API_KEY` → 一组**共享账号密码**（好记的密码短语）替代，作为唯一鉴权凭据。
- 新设备**自助注册**：安装脚本用 hostname 生成建议 `device_id`，用户确认/手改，
  带密码连接，服务器鉴权通过后自动注册。
- `DEVICES` 从硬编码字典 → **JSON 文件持久化**，运行时可增删。
- **设备数上限**（可配置），达到上限拒绝新设备注册、老设备照常重连。
- 现有已鉴权网页上加一个**设备管理小面板**（列表 + 删除，复用同一把密码）。

**不做**（留到下一轮）：多用户 / 账号体系 / 用户间隔离、密码找回、每设备独立密码、
token 轮换、审计日志、迁移到 SQLite/PostgreSQL、专门的远程管理后台、密码哈希存储、
登录失败限速。**如实说明**：用可记忆密码代替长随机 `API_KEY` 本身就是一次主动的
安全性换可用性取舍（熵更低），哈希与限速的缺失是在此取舍之上的**已知残留风险**，
接受前提是"小规模、可信环境、不面向公网大规模暴露"；详见"组件设计 1. 鉴权"一节。

## 组件设计

### 1. 鉴权：共享密码替代 `API_KEY`

- 语义上等价于把现有的 `API_KEY` 换成一个**人类可记忆的密码**，机制不变：
  HTTP 用 `X-API-Key`（值即密码）、WebSocket 用同名 header 校验，仍用
  `hmac.compare_digest()` 常量时间比对。
- 前提防护（本轮沿用/确认）：**全程 HTTPS/TLS**（生产已具备）、密码不进 URL、
  不写日志。
- **密码存储形态本轮已定，不留给实现阶段自由裁量：明文配置项**（环境变量 /
  `.env`，读取方式与现有 `API_KEY` 完全一致），**不引入哈希**。这与"范围"一节
  "不做密码哈希存储"的结论一致——哈希是为"服务端数据库泄露后仍需保护密码"这种
  场景设计的，本项目的密码本来就等价于一把共享密钥（类似现有 `API_KEY`），配置
  文件泄露本身就已经是失守，哈希在这个模型下不解决实际问题，属于过度设计。
- **关于"密码"比长随机 `API_KEY` 更弱这件事，如实说明，不回避**：人类可记忆的
  密码熵天然低于长随机串，这是用户在本轮设计一开始就明确选择的可用性 / 安全性
  取舍（好记 > 抗暴力破解），不是本文档意外引入的降级。缓解手段：① 密码仍应是
  一个有一定长度的短语，不是纯数字/常见词；② 全程 TLS + 不写日志，防止密码在
  传输链路上被动泄露；③ 无限速属于**已知的、本轮接受的残留风险**（服务不对公网
  做大规模暴露，威胁模型是"小规模、可信环境"），若未来暴露面变大或对外开放，
  登录限速应作为下一轮的优先项，而不是继续无限期搁置。

### 2. 设备清单：JSON 文件持久化

- `DEVICES` 从 `app.py` 里的硬编码字典改为一个**简单 JSON 文件**（如
  `server/devices.json`）。**不单独维护"显示名"字段**——用户在注册确认步骤里
  敲定的就是 `device_id` 本身，直接拿来展示，避免维护两份可能不一致的名字。
  每条记录字段：`device_id`、`created_at`、`last_active`。
- 服务器**启动时加载**该文件；**注册 / 删除时原子写入**（写临时文件再 rename）。
- **在线连接状态仍然只存内存**（现有 `AgentConnections.websockets`），不落盘——
  在线与否是瞬时状态，JSON 文件只存"有哪些设备被注册过"。
- **并发控制**：服务进程本身就固定单 worker（`--workers 1`，见现有部署约束），
  因此进程内用一把 `asyncio.Lock` 串行化"读取-判断上限-修改-写入"这个完整流程即可，
  不需要跨进程文件锁。**注册、删除、WebSocket 断开时更新 `last_active` 这三类写入
  必须持同一把锁执行**（断开事件不能绕过锁直接写盘），避免竞态下重复计数、漏判
  上限、或互相覆盖对方刚写入的内容。
- **`last_active` 更新时机**：只在**注册成功**和 **WebSocket 断开**两个事件时更新
  并写盘（同样经过上面那把锁），不做连接期间的持续写入（避免高频磁盘 I/O）；
  显示"在线"与否仍以内存中的 `websockets` 字典为准，`last_active` 只是"最近一次
  活跃"的参考时间戳。
- **文件损坏兜底**：启动时若 JSON 解析失败，视为空设备清单启动（不阻塞服务），
  并在启动日志里明确告警，避免一次损坏导致服务整体起不来。

### 3. 新设备自助注册

**协议形态（本轮锁定，不留给实现阶段自由发挥）**：注册是一个独立的**鉴权 HTTP
接口** `POST /api/devices/register`（请求体 `{"device_id": "..."}`，`X-API-Key`
即密码），成功返回 `200`（含设备信息），失败按下方错误处理返回明确状态码。
**WebSocket 连接不做注册**——`/ws/agent` 只接受"已经在设备清单里"的 `device_id`，
未注册的 `device_id` 直接拒绝（`1008`）。这样错误语义天然是 HTTP 状态码，不会
出现"WebSocket 握手里怎么表达 403"这种含糊情况。

安装脚本（各平台 Agent 的首次配置流程）：

1. **设备身份必须稳定、持久化在本地**：脚本首次运行时检查本地配置文件里有没有
   已保存的 `device_id`。**有 → 直接复用，跳过下面的建议/确认步骤**（重装、改
   hostname、迁移用户名都不会导致设备"看起来像新设备"）。
2. **没有**已保存的 `device_id` 时，才读取本地 **hostname** 作为建议值（**不用
   登录用户名**——用户名更容易带出个人信息）。
3. 终端展示建议名，用户**直接回车确认**或**手动改**成想要的名字。
4. 脚本调用 `POST /api/devices/register` 完成注册；**成功后把最终确定的
   `device_id` 写入本地配置文件持久化保存**，后续启动直接读取，不再重新生成。
5. 注册成功后，脚本用同一个 `device_id` 建立 `/ws/agent` 连接。

**服务端最小约束**（防止清单变脏）：

- `device_id` **规范化**：统一转小写，只允许 **3~32 位**字母 / 数字 / 连字符（`-`）。
- **查重**：已存在的 `device_id` 调用注册接口，视为**同一设备重连确认**（不新建
  条目，直接返回已有记录）；WebSocket 连接时同名顶替旧连接（现有行为不变）。
- **已知残留风险，明确接受**：两台不同物理设备如果都用默认建议名且都直接回车
  确认（比如都叫 `macbook-pro`），后连接的会顶替先连接的。这是"共享密码、无
  强身份"模型下的固有取舍，缓解手段是第 3 步"展示建议名并允许修改"这个人工确认
  环节，不额外加机器判重机制（比如 MAC 地址等），避免过度设计。

### 4. 设备数上限

- 一个**可配置上限**（如环境变量 `MAX_DEVICES`，默认值由实现阶段定）。
- 达到上限时的行为（关键，必须按此设计以免反噬）：
  - **已注册设备允许正常重连**；
  - **新 `device_id` 一律拒绝注册**，返回明确错误提示（"已达设备数上限"）；
  - **绝不"顶掉最旧设备"**——否则攻击者可持续注册把真实设备挤出去。
- 收益：即使密码泄露，攻击者也无法无限刷垃圾 `device_id`，破坏规模被封顶。
- 改上限需要重启或安全热加载，属于管理员操作。

### 5. 设备管理面板（复用现有鉴权）

- `server/static/index.html` 本身是**公开页面**（`GET /` 不鉴权，任何人都能打开），
  真正的鉴权发生在页面里发起的每一次 API 调用（用户在页面里填入密码，随请求带上
  `X-API-Key`）。本节说的"复用现有鉴权"指的是这个模式，不是说页面本身需要登录。
- 在这个页面上加一个**"设备管理"小面板**：
  - 列出已注册设备（含最后活跃时间 / 在线状态）；
  - 每条一个**删除按钮**（带二次确认）。
- 列表接口：`GET /api/devices`，`X-API-Key` 鉴权，返回数组，每项合并 JSON 里的
  持久字段与内存里的实时在线状态：
  ```json
  [{"device_id": "win-fukuoka", "created_at": "...", "last_active": "...", "online": true}]
  ```
- 删除接口：`DELETE /api/devices/{device_id}`，请求头带 `X-API-Key`（与其余接口
  一致，不用 query string 传密码，避免出现在服务器访问日志里）。用 `DELETE` 而非
  `GET`，天然避免"点开一个链接就误删"的 CSRF 式误触发。
- **删除的语义边界，必须明确写清楚，避免被误当成安全控制**：
  - 若被删设备当前在线，删除时**立即关闭其 WebSocket 连接**（不能只删 JSON 记录、
    留着一个"僵尸在线连接"）。
  - 删除只是**从设备清单里移除一条记录、腾出上限名额**，**不是**吊销该设备的
    访问权限——只要密码没换，被删设备的安装脚本重新跑一次注册接口，立刻能重新
    注册回来。真正的访问控制边界是共享密码本身，删除是"清单卫生工具"，不是
    "踢人下线的安全操作"。
- 不引入任何新的鉴权 / 管理体系，只是多两个受保护接口（列表 + 删除）+ 一点前端 UI。

## 数据流

```
新设备接入:
  安装脚本: 本地已存 device_id? --是--> 直接复用 --------------------------+
            |否                                                          |
            读 hostname 建议名 -> 用户确认/改名 -> device_id              |
                                                                          v
  安装脚本 --(POST /api/devices/register, device_id + 密码, TLS)--> 服务器
     服务器: 校验密码 -> 规范化 device_id -> 查重 (加锁串行执行)
        已存在? -> 200, 视为重连确认, 不新增记录
        新设备 & 未达上限? -> 200, 写入 devices.json (last_active=now)
        新设备 & 已达上限? -> 403, "已达设备数上限"
  安装脚本 <-- 200 --  把最终 device_id 写入本地配置持久化保存
  安装脚本 --(device_id, WS 连接 /ws/agent, 密码)--> 服务器建立长连接

日常发送 (不变):
  浏览器 --(target + text + 密码)--> /api/send --> 转发给对应在线 device

设备管理:
  浏览器(公开页面, 页面内填密码) --(密码)--> GET/DELETE /api/devices/* --> 读/删 devices.json
                                              (删除若设备在线, 同时关闭其 WS 连接)
```

## 现有设备迁移路径

现有两台设备 `win-fukuoka` / `mac-china` 不能失联，因此**必须有一段新旧凭据并存的
过渡期**，按以下顺序执行，每步之间互不阻塞、可独立验证：

1. **部署新服务端代码**，同时接受**旧 `API_KEY` 和新密码**两种凭据（过渡期内
   `check_api_key()` 对两者都放行，任一匹配即可）；`devices.json` 启动时**预置**
   `win-fukuoka` / `mac-china` 两条记录（内容等价于现在硬编码的 `DEVICES` 字典），
   两台设备**此时无需改动、无需重新注册**，用旧凭据继续连接不受影响。
2. **更新 Windows 端**本地配置里的凭据（旧 key → 新密码），重启 Agent，确认
   `/api/status` 显示 `win-fukuoka` 在线。
3. **更新 Mac 端**本地配置里的凭据，重启 Agent，确认 `/api/status` 显示
   `mac-china` 在线。
4. **两台都确认用新密码在线后**，服务端关闭旧 `API_KEY` 的接受（去掉双凭据分支，
   只保留新密码），完成迁移。
5. 过渡期时长没有硬性限制——只要第 2、3 步没做完，就不进入第 4 步；这是一个
   "确认后再收紧"的顺序依赖，不是定时任务。

## 错误处理

- 密码缺失 / 错误：沿用现有 `401`（HTTP）/ `1008`（WebSocket 关闭码）语义，
  `POST /api/devices/register`、`GET/DELETE /api/devices/*` 同样适用。
- `device_id` 不合法（不满足规范化规则）：`400`。
- 达到设备上限的新设备注册：`403` + "已达设备数上限" 文案，与 `401`（密码错误）
  明确区分，不能混为一谈。
- `/ws/agent` 连接一个**未注册**的 `device_id`：`1008`（关闭码不变，但含义从
  "不在硬编码白名单里"变成"没有先调用注册接口"）。
- 设备管理删除接口：密码校验失败 `401`；删除不存在的 `device_id` 返回 `404`
  （不做成静默幂等成功，避免前端误以为删除生效）。

## 测试与验证

- **服务端**：扩展 `server/tests/`，覆盖——
  - 正确 / 错误密码的鉴权（含过渡期新旧凭据双通道）；
  - `POST /api/devices/register`：新设备写入 JSON、重复 `device_id` 视为重连
    确认不新增、非法 `device_id` 返回 `400`、达到上限返回 `403`；
  - 并发注册（模拟两个几乎同时的注册请求）不突破上限、不产生重复/丢失记录；
  - `/ws/agent` 拒绝未注册的 `device_id`；
  - 删除接口：删除在线设备会断开其 WS 连接；删除不存在的设备返回 `404`；
  - `last_active` 只在注册成功 / 断开连接时更新。
- **前端**：设备管理面板列表 / 删除，靠手动验证。
- **端到端**：由用户在真实 Windows / Mac 上跑安装脚本自助注册 + 收发消息验证，
  并验证"重装/重启脚本后复用已保存 device_id、不重新生成"。

## 交付流程（用户指定）

1. 从 `main` 拉专用分支。
2. 写实现计划（只锁边界，具体实现交给 Codex）。
3. 脏活交给 Codex 执行；再由独立的 Codex 审查；按意见修改，循环到通过。
4. commit 信息用**中文**，不加 `Co-Authored-By` trailer（以用户身份提交）。
5. 用户在真实 Windows / Mac 上亲自测试自助注册与收发。
6. 用户确认无误并**明确许可**后，才合并回 `main`、删分支。

## 验收标准

- 用好记的密码（非长随机串）即可完成鉴权，替代原 `API_KEY`。
- 新设备跑安装脚本：读 hostname 建议名 → 用户确认/改 → 调用注册接口接入，无需
  人工预先在服务器配置设备；device_id 确认后持久化在本地，重装/重启脚本直接复用。
- 重复 / 非法 `device_id` 被正确处理（重连确认不新增、非法拒绝 `400`）。
