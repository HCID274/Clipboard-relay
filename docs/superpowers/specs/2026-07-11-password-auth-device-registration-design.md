# 密码鉴权与设备自助注册 — 设计文档

## 背景与问题

当前系统用**单一全局 `API_KEY`**（一段很长的随机串）做鉴权：HTTP 接口靠 `X-API-Key`
校验，WebSocket `/ws/agent?device_id=xxx` 用同一把 key 校验，设备白名单
（`win-fukuoka` / `mac-china`）**硬编码**在 `server/app.py` 的 `DEVICES` 字典里。

两个痛点：

1. **长 token 难记、配置麻烦**：新设备接入要手动复制粘贴一长串 key，还要有人先在
   服务器代码里把设备名加进 `DEVICES` 字典并重新部署，做不到"一步到位"。
2. **无法自助接入**：来一台新 Mac / Windows，没有脚本能"装完就自动连上并出现在
   设备列表里"。

目标：**用好记的密码代替长 token**，并让**新设备跑个安装脚本就能自解释注册接入**，
同时对"密码泄露后被批量刷垃圾设备"做一层轻量防护。

## 第一性原理判断（为什么是这个方案）

这一节记录设计过程中被逐条否掉的"更复杂方案"，说明为什么最终收敛到最简形态。

- **不需要 token 层**。密码每次随 TLS 请求发送、服务端直接比对即可。"登录换长期
  token"要多一套接口、多一份状态、多一条故障路径；在单人/几台设备场景下，一旦
  每台设备用的是共享密码，密码本身就已经在履行 API Key 的职责，再套一层 token
  属于过度设计。
- **不需要一次性配对码**。配对码防的是"密码泄露后被顺手注册新设备"，但这个风险在
  可信小规模场景可接受，且已由下面的"设备数上限"兜底，不值得为它增加一步手动输入。
- **不需要用户表 / 多用户隔离**。当前只有一个使用者，用户表是为将来"开放给别人用"
  预留的，属于 YAGNI。**多用户隔离明确推到下一轮**，真要开放时再引入账号体系。
- **不需要每台设备独立密码**。共享一套账号密码即可；唯一代价是"丢一台设备只能改
  共享密码、其余设备重新登录"，这个代价对个人几台设备完全可以接受，不值得为"单
  设备撤销"去维护设备密码表。
- **`device_id` 是非鉴权的路由标识**，不是凭据。它只回答"这条消息发给哪台设备"，
  鉴权完全由共享密码把关，因此设备可以自解释上报自己的名字。

## 范围

**做**：
- 全局 `API_KEY` → 一组**共享账号密码**（好记的密码短语）替代，作为唯一鉴权凭据。
- 新设备**自助注册**：安装脚本用 hostname 生成建议 `device_id`，用户确认/手改，
  带密码连接，服务器鉴权通过后自动注册。
- `DEVICES` 从硬编码字典 → **JSON 文件持久化**，运行时可增删。
- **设备数上限**（可配置），达到上限拒绝新设备注册、老设备照常重连。
- 现有已鉴权网页上加一个**设备管理小面板**（列表 + 删除，复用同一把密码）。

**不做**（留到下一轮）：多用户 / 账号体系 / 用户间隔离、密码找回、每设备独立密码、
token 轮换、审计日志、迁移到 SQLite/PostgreSQL、专门的远程管理后台。

## 组件设计

### 1. 鉴权：共享密码替代 `API_KEY`

- 语义上等价于把现有的 `API_KEY` 换成一个**人类可记忆的密码**，机制不变：
  HTTP 用 `X-API-Key`（值即密码）、WebSocket 用同名 header 校验，仍用
  `hmac.compare_digest()` 常量时间比对。
- 前提防护（本轮沿用/确认）：**全程 HTTPS/TLS**（生产已具备）、密码不进 URL、
  不写日志。
- 密码存储形态（明文配置项 vs 哈希）由实现阶段（Codex）决定，倾向沿用现有
  `API_KEY` 的读取方式（环境变量 / `.env`），改动面最小。

### 2. 设备清单：JSON 文件持久化

- `DEVICES` 从 `app.py` 里的硬编码字典改为一个**简单 JSON 文件**（如
  `server/devices.json`），字段至少含：`device_id`、显示名、创建时间、
  最后活跃时间。
- 服务器**启动时加载**该文件；**注册 / 删除时原子写入**（写临时文件再 rename）。
- **在线连接状态仍然只存内存**（现有 `AgentConnections.websockets`），不落盘——
  在线与否是瞬时状态，JSON 文件只存"有哪些设备被注册过"。

### 3. 新设备自助注册

安装脚本（各平台 Agent 的首次配置流程）：

1. 读取本地 **hostname** 作为建议 `device_id`（**不用登录用户名**——用户名更容易
   带出个人信息）。
2. 终端展示建议名，用户**直接回车确认**或**手动改**成想要的名字。
3. 脚本带着 `device_id` + 密码向服务器发起连接（WebSocket 握手，或一个专门的
   注册接口 —— 具体形态由实现阶段决定）。
4. 服务器校验密码通过后，把 `device_id` **自动注册**进设备清单，无需人工预先配置。

**服务端最小约束**（防止清单变脏）：

- `device_id` **规范化**：统一转小写，只允许 **3~32 位**字母 / 数字 / 连字符（`-`）。
- **查重**：已存在的 `device_id` 视为**同一设备重连**（替换旧连接），**不新建条目**。

### 4. 设备数上限

- 一个**可配置上限**（如环境变量 `MAX_DEVICES`，默认值由实现阶段定）。
- 达到上限时的行为（关键，必须按此设计以免反噬）：
  - **已注册设备允许正常重连**；
  - **新 `device_id` 一律拒绝注册**，返回明确错误提示（"已达设备数上限"）；
  - **绝不"顶掉最旧设备"**——否则攻击者可持续注册把真实设备挤出去。
- 收益：即使密码泄露，攻击者也无法无限刷垃圾 `device_id`，破坏规模被封顶。
- 改上限需要重启或安全热加载，属于管理员操作。

### 5. 设备管理面板（复用现有鉴权）

- 在现有已鉴权页面 `server/static/index.html` 上加一个**"设备管理"小面板**：
  - 列出已注册设备（含最后活跃时间 / 在线状态）；
  - 每条一个**删除按钮**（带二次确认）。
- 删除走一个**受同一把共享密码保护的 HTTP 接口**，直接增删 JSON 文件里的条目——
  **不引入任何新的鉴权 / 管理体系**，只是多一个受保护接口 + 一点前端 UI。
- 用途：清理误注册 / 垃圾条目，给真实设备腾出上限名额。删除入口做成网页按钮而非
  "只能本地命令行"，避免管理体验难受。

## 数据流

```
新设备接入:
  安装脚本 --(读 hostname, 用户确认/改名)--> device_id
  安装脚本 --(device_id + 密码, TLS)--> 服务器
     服务器: 校验密码 -> 规范化 device_id -> 查重
        已存在? -> 视为重连, 替换连接
        新设备 & 未达上限? -> 写入 devices.json, 建立连接
        新设备 & 已达上限? -> 拒绝, 返回"已达上限"

日常发送 (不变):
  浏览器 --(target + text + 密码)--> /api/send --> 转发给对应在线 device

设备管理:
  浏览器(已鉴权页面) --(密码)--> 设备管理接口 --> 读/删 devices.json
```

## 现有设备迁移路径

- 现有两台设备 `win-fukuoka` / `mac-china` 不能失联。
- 迁移时把这两条**预置进 `devices.json`**（等价于现在硬编码字典的内容），
  并把服务端鉴权从旧 `API_KEY` 切到新密码。
- 两台设备各自更新本地配置里的凭据（旧 key → 新密码）后重连即可。
- 过渡细节（是否需要一段"新旧凭据并存期"避免同时失联，还是直接切换）由实现
  阶段（Codex）与用户手动测试环节把控。

## 错误处理

- 密码缺失 / 错误：沿用现有 `401`（HTTP）/ `1008`（WebSocket 关闭码）语义。
- 达到设备上限的新设备注册：明确的 `403`（或约定状态码）+ "已达设备数上限" 文案，
  不能和"密码错误"混为一谈。
- 设备管理删除接口：密码校验失败同样 `401`；删除不存在的 `device_id` 幂等处理。

## 测试与验证

- **服务端**：扩展 `server/tests/`，覆盖——
  - 正确 / 错误密码的鉴权；
  - 新设备注册写入 JSON；重复 `device_id` 视为重连不新增；
  - `device_id` 规范化与非法字符拒绝；
  - 达到上限时拒绝新设备、老设备仍可重连；
  - 删除接口增删 JSON 生效。
- **前端**：设备管理面板列表 / 删除，靠手动验证。
- **端到端**：由用户在真实 Windows / Mac 上跑安装脚本自助注册 + 收发消息验证。

## 交付流程（用户指定）

1. 从 `main` 拉专用分支。
2. 写实现计划（只锁边界，具体实现交给 Codex）。
3. 脏活交给 Codex 执行；再由独立的 Codex 审查；按意见修改，循环到通过。
4. commit 信息用**中文**，不加 `Co-Authored-By` trailer（以用户身份提交）。
5. 用户在真实 Windows / Mac 上亲自测试自助注册与收发。
6. 用户确认无误并**明确许可**后，才合并回 `main`、删分支。

## 验收标准

- 用好记的密码（非长随机串）即可完成鉴权，替代原 `API_KEY`。
- 新设备跑安装脚本：读 hostname 建议名 → 用户确认/改 → 自动注册接入，无需人工
  预先在服务器配置设备。
- 重复 / 非法 `device_id` 被正确处理（重连不新增、非法拒绝）。
- 设备清单持久化在 JSON 文件，重启不丢。
- 达到设备上限时新设备被拒、老设备正常重连，真实设备不会被挤出。
- 已鉴权网页上可查看并删除设备。
- 现有两台设备平滑迁移，不失联。
- 服务端新增测试全部通过；发送流程无回归。
