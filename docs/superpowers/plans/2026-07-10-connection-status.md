# 连接状态可视化 —— 实现计划

对应设计文档：`docs/superpowers/specs/2026-07-10-connection-status-design.md`

> 本计划只锁定接口契约和边界，不预写实现代码——具体实现（变量命名、CSS 写法、
> 测试组织方式）由 Codex 决定并完成，Claude 负责出方向、审结果。

## 任务 1：服务端新增 `GET /api/status`

**边界**：
- 文件：`server/app.py`（新增路由，不改动现有路由行为）
- 鉴权：复用现有 `check_api_key()`，语义与 `/api/send` 一致（缺/错密钥 401，未配置 500）
- 返回契约（200）：`{"devices": {"win-fukuoka": bool, "mac-china": bool}}`，键来自 `DEVICES`
- 纯读取 `agents.websockets`，无副作用，不改动任何连接状态
- 顺带建立 `server/tests/`（当前无测试），覆盖：无密钥/错密钥 401；`API_KEY`
  未配置时 500；默认全离线；模拟一个 Agent 连接后对应设备变 `true`，断开后
  变回 `false`；两个设备同时在线时映射同时反映两者状态
- 测试框架/依赖版本、pyproject.toml 怎么改由 Codex 决定

## 任务 2：前端连接状态可视化

**边界**：
- 文件：`server/static/index.html`（单文件，保持内联 CSS/JS 现状，不引入构建工具）
- 轮询 `/api/status`：密钥非空时才轮询；进页面、切换目标设备时各查一次，
  之后约 4 秒一次
- 三态展示（选中设备下方一个状态指示）：
  - 服务端不可达（fetch 异常，或收到 401/500 以外的非 200 状态码）
  - 密钥错误（401）
  - 服务端配置错误（500，`API_KEY` 未配置）——需要单独文案，不能和"密钥错误"混淆
  - 客户端离线 / 已连接（200 + 对应布尔值）
- **连接状态展示必须使用独立于 `#status` 的新 DOM 元素**，不能复用发送结果的
  展示区域——两者语义不同，复用会导致互相覆盖
- 下拉框每个选项用**文字后缀**标注在线/离线（如"（在线）/（离线）"），
  **不用颜色**——手机原生 `<option>` 无法可靠上色
- 发送逻辑不变，不因设备显示离线而拦截发送按钮
- 具体 DOM 结构、CSS 类名、JS 组织方式由 Codex 决定
